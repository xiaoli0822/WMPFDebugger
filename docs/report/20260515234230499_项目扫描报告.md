# WMPFDebugger 项目扫描报告

生成时间：2026-05-15 23:42:30.499（Asia/Shanghai）

## 结论

这是一个面向 Windows 的 WeChat Mini Program 调试工具，核心链路是 Electron GUI + TypeScript 主进程服务 + Frida 注入 + WebSocket/CDP 代理。当前仓库可直接编译，基础验证已通过。

## 项目轮廓

- `src/index.ts`：CLI 入口，启动 debug server / proxy server，并注入 `frida/hook.js`
- `electron/main.ts`：Electron 主进程，创建窗口、处理 IPC、管理设置与外部浏览器打开
- `electron/debugService.ts`：核心服务层，负责 Frida attach、进程扫描、Hook 插件生命周期、CDP 转发
- `electron/hookManager.ts`：Hook 插件管理、持久化与组合脚本生成
- `renderer/`：窗口 UI、状态展示、插件管理、进程扫描
- `frida/hook.js`：运行时注入脚本
- `frida/config/addresses.*.json`：按 WMPF 版本区分的地址配置

## 技术栈

- Electron 33
- TypeScript 5.9
- Frida 17
- `ws` / `protobufjs`
- Node.js LTS v22+

## 脚本

- `npm run cli`
- `npm run build:electron`
- `npm run start`
- `npm run dev`
- `npm test`

## 本地验证

- `cmd /c npm test`：通过
- `cmd /c npm run build`：通过

## 当前风险

- 工作区本身已有多处未提交改动，包含 `electron/*`、`renderer/*`、`src/index.ts`、`tsconfig*.json`、`frida/hook.js` 等
- 存在已删除但仍被 Git 跟踪的 `.aone_copilot/*` 文件，以及新出现的 `.npmrc`
- 仓库里同时存在 `package-lock.json` 与 `yarn.lock`，包管理策略不统一
- 该工具依赖本机 `WeChatAppEx.exe`、对应版本配置以及 Frida 权限，普通构建无法覆盖真实注入链路
