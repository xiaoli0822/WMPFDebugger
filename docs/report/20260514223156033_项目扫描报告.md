## 项目结构

- `package.json`：项目脚本、依赖与 Electron 入口配置，`main` 指向 `dist-electron/main.js`。
- `electron/main.ts`：Electron 主进程，创建窗口、注册 IPC、管理浏览器打开 DevTools。
- `electron/preload.ts`：通过 `contextBridge` 暴露 `debuggerAPI` 和 `windowAPI`。
- `electron/debugService.ts`：GUI 调试服务核心，负责 Frida attach、Debug WebSocket、CDP Proxy、Hook 插件注入、日志与状态事件。
- `electron/hookManager.ts`：Hook 插件增删改、启停、持久化和脚本组合。
- `renderer/index.html` / `renderer/renderer.js` / `renderer/styles.css`：Electron 渲染层 UI、状态流、插件管理、进程扫描和日志展示。
- `src/index.ts`：CLI 原始入口，启动 debug server、CDP proxy，并注入 `frida/hook.js`。
- `src/third-party/*`：远程调试协议相关第三方/提取代码。
- `frida/hook.js`：运行时 Frida patch 逻辑。
- `frida/config/addresses.*.json`：WMPF 版本偏移配置，共识别 26 个版本配置。
- `README.md` / `README.zh.md` / `ADAPTATION.md` / `EXTENSION.md`：使用、适配和扩展调试文档。

## 风险清单

- R1：根 `tsconfig.json` 全仓类型检查失败；如果 CI 或开发者使用该配置，会被 Electron 类型库 DOM 依赖阻断。
- R2：项目同时存在 `package-lock.json` 和 `yarn.lock`，README 又推荐 `yarn`，依赖锁定策略不统一。
- R3：`.aone_copilot` 当前既被代码用作运行态配置目录，又存在已跟踪文件删除状态；需要明确它是用户配置目录还是仓库文档/计划资产。
- R4：核心功能依赖本机存在 `WeChatAppEx.exe` 进程、匹配版本的 `frida/config/addresses.<version>.json`，且 Frida attach 需要相应权限；普通构建验证无法覆盖真实注入链路。
- R5：Electron 主进程会保存浏览器路径到 `.aone_copilot/settings.json`，当前 `.gitignore` 未忽略 `.aone_copilot`，后续可能误提交本机路径或 Hook 脚本配置。
- R6：`src/third-party` 明确来源于微信开发者工具提取代码，许可证和分发边界需要持续关注。
- R7：`.npmrc` 中 `electron_mirror`、`disturl` 在 npm 11 下有 unknown config 警告，后续 npm 主版本可能停止兼容。

## 建议下一步

1. 统一包管理器：保留 `yarn.lock` 或 `package-lock.json` 之一，并同步 README/脚本说明。
2. 修复根 `tsconfig.json`：增加 `skipLibCheck` 或拆分 CLI/Electron 类型检查边界，避免误用根配置失败。
3. 明确 `.aone_copilot` 策略：如果是运行态配置，加入 `.gitignore` 并迁移已有文档；如果是项目资产，避免运行态配置写入同目录。
