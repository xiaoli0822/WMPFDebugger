# WASM/内存 Hook 示例与排障

## 目标

补一份面向插件作者的最小可落地说明，覆盖：

- WASM Hook 的基本写法
- 内存 Hook 的基本写法
- 幂等写法，避免重复注入或重复 patch
- 常见故障的定位顺序

## 使用前提

- 插件脚本运行在页面上下文，可能随着 DevTools 重开、页面刷新、后端重连而再次注入。
- 同一段 Hook 若没有做幂等保护，最常见问题是重复包裹原函数、重复 `Interceptor.attach`、重复改写内存。
- 禁用或删除插件后，已经作用到当前页面运行时的 monkey patch 不会自动回滚；通常需要刷新目标页或重开 DevTools。

## 示例一：WASM 导出函数 Hook

适用场景：

- 需要观测 `WebAssembly.instantiate` / `instantiateStreaming`
- 需要在模块实例创建后替换导出函数

```js
(function wasmHookExample() {
    const guardKey = "__wmpf_hook_wasm_example__";
    if (globalThis[guardKey]) {
        console.info("[hook] WASM 示例已安装，跳过重复注入");
        return;
    }
    globalThis[guardKey] = true;

    const originalInstantiate = WebAssembly.instantiate;

    WebAssembly.instantiate = async function patchedInstantiate(bufferSource, importObject) {
        const result = await originalInstantiate.call(this, bufferSource, importObject);
        const instance = result && result.instance ? result.instance : result;

        if (instance && instance.exports && typeof instance.exports.encrypt === "function") {
            if (!instance.exports.encrypt.__wmpfWrapped) {
                const originalEncrypt = instance.exports.encrypt;
                function wrappedEncrypt(...args) {
                    console.info("[hook] wasm encrypt args", args);
                    const value = originalEncrypt.apply(this, args);
                    console.info("[hook] wasm encrypt result", value);
                    return value;
                }
                wrappedEncrypt.__wmpfWrapped = true;
                instance.exports.encrypt = wrappedEncrypt;
            }
        }

        return result;
    };
})();
```

关键点：

- 先用 `globalThis[guardKey]` 做一次全局幂等保护。
- 再给具体导出函数打 `__wmpfWrapped` 标记，避免实例复用时二次包裹。
- 所有 patch 都尽量保留原函数引用，便于排障时快速对比。

## 示例二：Frida 内存 Hook

适用场景：

- 需要对 native 函数入参/返回值做观测
- 需要一次性 patch 某个地址或偏移

```js
const installedHooks = globalThis.__wmpfInstalledHooks || (globalThis.__wmpfInstalledHooks = new Set());

function attachOnce(tag, address, handlers) {
    if (installedHooks.has(tag)) {
        console.log("[hook] 已存在 Hook，跳过：", tag);
        return;
    }
    Interceptor.attach(ptr(address), handlers);
    installedHooks.add(tag);
    console.log("[hook] 已安装：", tag, address);
}

attachOnce("send-filter-enter", "0x12345678", {
    onEnter(args) {
        this.arg0 = args[0];
        console.log("[hook] onEnter arg0 =", this.arg0);
    },
    onLeave(retval) {
        console.log("[hook] onLeave retval =", retval);
    },
});
```

关键点：

- 不要直接裸调多次 `Interceptor.attach`，优先封装成 `attachOnce`。
- `tag` 建议稳定且可读，例如模块名 + 偏移 + 目的。
- 如果地址会随版本变化，优先把“地址解析”和“attach”拆开，日志里同时打印版本、模块基址、偏移。

## 示例三：幂等内存写入

适用场景：

- 只需要把某个字节位改成目标值
- 后端重连、脚本重复执行时不能重复破坏现场

```js
const patchedBytes = globalThis.__wmpfPatchedBytes || (globalThis.__wmpfPatchedBytes = new Map());

function writeU8Once(tag, address, nextValue) {
    const target = ptr(address);
    const currentValue = target.readU8();

    if (patchedBytes.has(tag)) {
        console.log("[patch] 已处理过，跳过：", tag, "当前值 =", currentValue);
        return;
    }

    patchedBytes.set(tag, currentValue);
    target.writeU8(nextValue);
    console.log("[patch] 已写入：", tag, "old =", currentValue, "new =", nextValue);
}

writeU8Once("filter-flag", "0x12345680", 1);
```

建议：

- 首次写入前缓存原值，至少便于日志复盘和手工回滚。
- 如果后续确实要支持回滚，可再提供 `restorePatchedByte(tag, address)`。
- 对多字节 patch 同样建议先缓存原始 `ByteArray`。

## 推荐幂等模式

### 1. 全局开关

适合“整段脚本只允许安装一次”的场景。

```js
if (globalThis.__myHookInstalled) {
    return;
}
globalThis.__myHookInstalled = true;
```

### 2. 函数包装标记

适合替换 `fetch`、`WebAssembly.instantiate`、导出函数等对象方法。

```js
if (!targetFn.__wrappedByWmpf) {
    const original = targetFn;
    function wrapped(...args) {
        return original.apply(this, args);
    }
    wrapped.__wrappedByWmpf = true;
    targetObject.method = wrapped;
}
```

### 3. Set/Map 注册表

适合管理多地址、多模块、多阶段 Hook。

```js
const registry = globalThis.__hookRegistry || (globalThis.__hookRegistry = new Set());
if (!registry.has(hookId)) {
    registry.add(hookId);
    // attach / write memory
}
```

## 常见故障排查

### 1. 日志显示插件已启用，但没有效果

优先检查：

- 插件是否是空脚本；空脚本会保持禁用。
- 是否在旧页面上下文里观察结果；很多 Hook 需要刷新页面或重开 DevTools 才会命中新上下文。
- 是否命中了错误版本地址；内存 Hook 先打印模块基址、偏移和最终地址。

### 2. 重开 DevTools 后效果重复、日志刷屏

大概率是脚本重复注入但没有幂等保护。检查：

- 是否缺少 `globalThis` 级 guard
- 是否重复 `Interceptor.attach`
- 是否对同一导出函数重复包裹

### 3. 打开页面立刻报 `TypeError` 或 `cannot redefine`

优先检查：

- 被替换对象是否只读或不可配置
- 包装前是否已经被别的脚本改写
- `Object.getOwnPropertyDescriptor` 是否允许重新定义

必要时先打印：

```js
console.log(Object.getOwnPropertyDescriptor(WebAssembly, "instantiate"));
```

### 4. 内存写入后程序异常或闪退

优先检查：

- 地址是否属于当前版本
- 写入宽度是否正确，例如 `writeU8` / `writeU32` / `writePointer`
- 是否误把代码段、只读段当成可写区域
- 是否在错误线程/错误时机提前 patch

建议先做只读观测，再做写入；先小范围验证，再扩大覆盖。

### 5. 删除或禁用插件后效果仍然存在

这是预期现象之一。原因通常不是“插件还在运行”，而是：

- 当前页面上下文已经完成 monkey patch
- native attach 已经挂上，当前进程生命周期内仍然生效

处理方式：

- 刷新目标页
- 重开 DevTools
- 必要时重启目标进程

## 最小排障清单

每次排查建议按这个顺序走：

1. 先确认插件脚本非空，且启用后没有被自动回退到禁用。
2. 打印一次“脚本已注入”日志，确认当前上下文确实执行到了入口。
3. 给关键 Hook 点加唯一 tag，确认有没有重复安装。
4. 对内存 Hook 打印版本、模块基址、偏移、最终地址。
5. 变更脚本后刷新页面或重开 DevTools，再看效果。

## 建议沉淀

- 需要长期维护的 Hook，统一封装 `attachOnce`、`writeOnce`、`wrapOnce` 三类助手。
- 每个插件头部固定输出插件名、版本、目标模块、关键地址，方便日志检索。
- 对风险较高的内存写入，先落只读观测版，再落写入版，避免一次性改坏现场。
